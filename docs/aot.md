# NativeAOT and trimming

The library is `IsAotCompatible=true` and `IsTrimmable=true`. The bundled Roslyn source generator (`LocalJob.SourceGenerator`) emits the `AddLocalJobs` extension method specific to your project, so DI registration does not rely on `Assembly.GetTypes()` reflection.

## Registration

```csharp
builder.Services.AddLocalJobs(builder.Configuration);
```

> **Build at least once.** The generator runs as part of compilation. Before the first build of your project, `AddLocalJobs` does not exist as a symbol and the IDE will report `CS1061: 'IServiceCollection' does not contain a definition for 'AddLocalJobs'`. Run `dotnet build` and the symbol resolves. From then on every build refreshes it. See [troubleshooting.md](troubleshooting.md#addlocaljobs-is-not-recognized--cs1061--no-definition-for-addlocaljobs) if it still doesn't.

The generator scans your compilation, finds every non-abstract subclass of `LocalBackgroundJob`, and emits an `internal` extension method into your assembly. Here is the actual output for the `samples/LocalJob.Sample` project (file `LocalJobGeneratedRegistration.g.cs`):

```csharp
// <auto-generated/>
#nullable enable

using Microsoft.Extensions.Configuration;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.DependencyInjection.Extensions;
using Microsoft.Extensions.Hosting;

namespace LocalJob;

[System.CodeDom.Compiler.GeneratedCode("LocalJob.SourceGenerator", "1.0.0")]
internal static class LocalJobGeneratedRegistration
{
    internal static IServiceCollection AddLocalJobs(this IServiceCollection services, IConfiguration? configuration = null)
    {
        services.ConfigureLocalJobOptions(configuration);
        services.TryAddEnumerable(ServiceDescriptor.Singleton<IHostedService, global::LocalJob.Sample.HeartbeatJob>());
        services.TryAddEnumerable(ServiceDescriptor.Singleton<IHostedService, global::LocalJob.Sample.MetricsFlushJob>());
        services.TryAddEnumerable(ServiceDescriptor.Singleton<IHostedService, global::LocalJob.Sample.TempCleanupJob>());
        return services;
    }
}
```

The class is `internal` so each consuming assembly gets its own copy with no risk of collision across projects.

The configuration binding inside `ConfigureLocalJobOptions` reads strings from `IConfigurationSection["..."]` and parses with `TimeSpan.TryParse` / `bool.TryParse`. No reflection, no `ConfigurationBinder.Bind`.

Generic job classes cannot be auto-registered (a `ServiceDescriptor` needs a closed type); the generator skips them and reports warning `LJOB001` with guidance instead of emitting uncompilable code.

To inspect the generator output for your own project, build with:

```sh
dotnet build -p:EmitCompilerGeneratedFiles=true -p:CompilerGeneratedFilesOutputPath=Generated
```

The file lands at `Generated/LocalJob.SourceGenerator/LocalJob.SourceGenerator.LocalJobGenerator/LocalJobGeneratedRegistration.g.cs`.

## Project setup for AOT-published apps

```xml
<PropertyGroup>
  <PublishAot>true</PublishAot>
  <InvariantGlobalization>true</InvariantGlobalization>
</PropertyGroup>
```

Build:

```sh
dotnet publish -c Release
```

The library itself adds zero AOT incompatibilities.

## ProjectReference vs PackageReference

When you install via NuGet, the generator DLL is delivered automatically through the `analyzers/dotnet/cs` folder of the package. Compilers running on your project pick it up.

When you reference the library via `<ProjectReference>`, analyzers do **not** flow automatically. Reference the generator project explicitly:

```xml
<ProjectReference Include="..\path\to\LocalJob.SourceGenerator.csproj"
                  OutputItemType="Analyzer"
                  ReferenceOutputAssembly="false" />
```

This is what the `samples/LocalJob.Sample` project does.
